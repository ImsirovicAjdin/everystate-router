# @everystate/router v1.1.0: Pass `view` in Boot Context

**Date:** March 24, 2026
**Type:** Non-breaking enhancement (additive)
**File changed:** `router.js`, `index.d.ts`

---

## Summary

The router now passes the route's `view` key to the `boot()` context object. Previously, `boot()` received `{ store, el, signal, params }`. It now receives `{ store, el, signal, params, view }`.

This is a one-line, non-breaking change. Existing `boot()` functions that don't destructure `view` are completely unaffected.

---

## Motivation

The `view` key is already defined on every route:

```js
{ path: '/data', view: 'data', component: DataPage }
```

And the router already computes it internally during navigation (as `viewKey` on line 162 of `router.js`). But it was never passed to `boot()`, which meant every boot function that needed a prefix had to hardcode it:

```js
const DataPage = {
  boot({ store, el }) {
    flatten(spec, store, 'view.data');  // <-- hardcoded, duplicates route.view
    return mount(store, 'view.data', el, handlers, components);
  }
};
```

This is redundant. The router knows the view name. The boot function shouldn't have to re-state it.

---

## The Change

### `router.js`

In the `navigate()` function, the call to `component.boot()` changes from:

```js
unboot = await component.boot({ store, el: root, signal, params: resolved.params });
```

to:

```js
unboot = await component.boot({ store, el: root, signal, params: resolved.params, view: viewKey });
```

`viewKey` is already computed on the same function scope (line 162: `const viewKey = resolved.view`). No new logic. No new variables. Just passing an existing value through.

### `index.d.ts`

The `BootContext` interface gains one optional field:

```ts
export interface BootContext {
  store: any;
  el: HTMLElement;
  signal: AbortSignal;
  params: Record<string, string>;
  view: string;  // NEW
}
```

---

## Why This Matters

This change is architecturally significant despite being tiny. It completes the boot context - the router now tells boot functions *everything* it knows about the current route. This enables `@everystate/view` (and any other renderer) to derive store prefixes from the route automatically, without hardcoding.

Concretely, this unblocks the `specPage()` utility in `@everystate/view` (see `@everystate/view` v1.2.0 release notes), which uses `view` to auto-derive the store prefix:

```js
const prefix = `view.${view}`;  // 'view.data', 'view.home', etc.
```

Without this change, every boot function must know its own prefix. With it, the prefix is derived from the route definition - the single source of truth.

---

## Impact on Existing Code

**None.** This is purely additive.

- Boot functions that destructure `{ store, el }` - unchanged, `view` is ignored.
- Boot functions that destructure `{ store, el, signal, params }` - unchanged, `view` is ignored.
- Boot functions that destructure `{ store, el, view }` - new capability, can use the view key.
- React/Vue/Angular/Solid boot functions - unaffected. They don't use `@everystate/view` and don't need `view`.

---

## Impact on Framework Adapters

| Package | Impact |
|---------|--------|
| `@everystate/react` | None. React boot functions use their own rendering. |
| `@everystate/vue` | None. Vue boot functions use their own rendering. |
| `@everystate/angular` | None. Angular boot functions use their own rendering. |
| `@everystate/solid` | None. Solid boot functions use their own rendering. |
| `@everystate/view` | Enabled. `specPage()` can now auto-derive prefixes. |

The router remains renderer-agnostic. It passes `view` as data. What any renderer does with it (or whether it ignores it) is entirely up to the renderer.

---

## Backward Compatibility

Fully backward compatible. No breaking changes. No new dependencies. No configuration changes.

The only observable difference: `boot()` receives one additional property in its context object. JavaScript ignores extra properties in destructuring, so no existing code can break.
